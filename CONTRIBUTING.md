# Гайд по совместной работе (Git workflow)

Как двое (и больше) разработчиков работают над этим проектом, не мешая друг другу.
Проектную документацию по коду смотри в [CLAUDE.md](CLAUDE.md).

## Ветки

| Ветка | Роль | Правило |
|-------|------|---------|
| `master` | Стабильная (main) — всегда рабочая версия | Пишем сюда **только** через перелив из `revision-2` |
| `revision-2` | Рабочая (develop) — сюда стекается вся текущая работа | Пишем сюда **только** через Pull Request из feature-веток |
| `feature/*`, `fix/*`, `chore/*` | Временные ветки под одну задачу | Живут пока задача не влита, потом удаляются |

**Золотое правило:** напрямую в `master` и `revision-2` не коммитим. Любое изменение проходит через отдельную ветку и Pull Request.

## Часть 1. Обычная работа над задачей (часто)

```bash
git checkout revision-2 && git pull      # 1. взять свежую рабочую ветку
git checkout -b feature/название-задачи   # 2. создать ветку под задачу
# ...пишем код, делаем коммиты...
git push -u origin feature/название-задачи # 3. отправить на GitHub
# 4. На GitHub: New Pull Request, base: revision-2 ← compare: feature/...
#    → второй человек делает ревью → Merge → Delete branch
```

Именование веток: `feature/...` — новая функциональность, `fix/...` — исправление, `chore/...` — рутина (доки, конфиги, зависимости).

## Часть 2. Перелив в master (реже, после проверок)

Когда `revision-2` протестирована и стабильна — переносим накопленное в `master`.

**Рекомендуемый способ — через Pull Request (с ревью):**
```
На GitHub: New Pull Request, base: master ← compare: revision-2
→ Create → проверить → Merge
```

**Либо локально:**
```bash
git checkout master && git pull
git merge revision-2
git push origin master
git checkout revision-2                    # вернуться на рабочую ветку
```

## Правило синхронизации

После перелива `master` == `revision-2` — ветки совпадают. Если что-то когда-нибудь попадёт **напрямую в `master`** (например, срочный фикс), сразу влейте `master` обратно в `revision-2`:

```bash
git checkout revision-2
git merge master
git push origin revision-2
```

**Изменения всегда текут в обе стороны** — так ветки не расходятся.

## Перед началом работы

1. `git checkout revision-2 && git pull` — всегда стартуй с последней версии, чтобы видеть чужие изменения.
2. Одну задачу — в одну ветку. Не смешивай несвязанные изменения в одном PR.
3. Если оба правите один файл — при слиянии возможен конфликт; Git подскажет, разрешается вручную (оставить нужные строки, убрать маркеры `<<<<<<<`).
