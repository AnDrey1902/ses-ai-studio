# Гайд по совместной работе (Git workflow)

Как небольшая команда (2–3 человека + AI-ассистенты) работает над этим проектом,
не мешая друг другу и не ломая рабочую версию.
Проектную документацию по коду смотри в [CLAUDE.md](CLAUDE.md).

## Модель ветвления — GitHub Flow

Одна постоянная ветка + короткоживущие ветки под задачи. Никаких `develop` / `revision-*`.

| Ветка | Роль | Правило |
|-------|------|---------|
| `master` | Главная, **неприкосновенная**. Всегда рабочий, протестированный код; из неё идёт деплой. | Пишем сюда **только** через Pull Request. Прямой push и `--force` запрещены (защищено на GitHub). |
| `feature/*`, `fix/*`, `chore/*` | Временные ветки под одну задачу. | Создаются от свежего `master`, вливаются через PR, после слияния **удаляются**. |

**Золотое правило:** напрямую в `master` не коммитим и не пушим. Любое изменение проходит
через отдельную ветку и Pull Request. Флаги `--force` / `-f` — **под запретом** (затирают чужую историю).

Именование веток: `feature/...` — новая функциональность, `fix/...` — исправление,
`chore/...` — рутина (доки, конфиги, зависимости). Название — короткое, на английском:
`feature/hero-redesign`, `fix/lead-modal-close`.

## Часть 1. Обычная работа над задачей

```bash
git checkout master && git pull        # 1. взять свежую главную ветку
git checkout -b feature/название-задачи # 2. создать ветку под задачу
# ...пишем код, делаем коммиты...
npm run build                          # 3. убедиться, что проект собирается
git push -u origin feature/название-задачи # 4. отправить ветку на GitHub
# 5. На GitHub: Create Pull Request, base: master ← compare: feature/...
#    → второй человек делает ревью → Merge → Delete branch
```

**После мержа PR — убери за собой** (чтобы локально не копились мёртвые ветки):

```bash
git checkout master && git pull   # подтянуть смёрженное
git fetch --prune origin          # убрать устаревшие origin/<ветка>
git branch -d feature/твоя-ветка  # удалить локальную (merged)
```

- **Одна сессия/задача — одна ветка — один PR.** Связанные подзадачи (например, правки в рамках одной сессии) держи в **одной** ветке и одном PR — не плоди отдельный PR на каждую мелочь. И наоборот: **не**связанные задачи не мешай в одном PR.
- **Сообщения коммитов — на английском**, в стиле Conventional Commits: `feat:`, `fix:`,
  `docs:`, `chore:`, `refactor:`, `style:` (напр. `feat: add payback calculator`).
- **Merge делает человек**, а не автор в одиночку и не AI-ассистент: PR → ревью → кнопка *Merge* на GitHub.
- Если `git branch -d` ругается «not fully merged» — PR смёржили режимом **Squash**; тогда безопасно удалять через `git branch -D` (содержимое уже в `master`).

## Быстрый режим (git-алиасы)

Чтобы не набирать всю цепочку вручную, рутина завёрнута в пять команд:

| Команда | Что делает |
|---------|-----------|
| `git start <ветка>` | Начало задачи: свежий `master` + `pull` + `prune` + новая ветка. |
| `git save "msg"` | `add -A` + `commit`. По ходу подзадач жми сколько нужно. |
| `git ship "msg"` | Коммит остатка + `push` + создать Pull Request. |
| `git land` | (соло) squash-merge PR + удалить ветку + вернуться на чистый `master`. |
| `git sync` | Подтянуть `master` + prune + снести локальные ветки, уже слитые и удалённые на GitHub. |

Типичная сессия: `git start feature/x` → работа, изредка `git save "…"` → `git ship "feat: x"` → `git land`.
Две команды по краям, между ними — просто работа. **Одна сессия = одна ветка = один PR** —
не по отдельному PR на каждую мелкую правку, а все связанные подзадачи в одной ветке.

**Разовая настройка** (один раз на машину):

1. Поставить GitHub CLI: `winget install --id GitHub.cli -e` (Windows).
2. Авторизовать: `gh auth login` → GitHub.com → HTTPS → *Login with a web browser*.
3. Вставить блок в свой глобальный `~/.gitconfig`:

```ini
[alias]
	start = "!f() { b=$(git symbolic-ref --short refs/remotes/origin/HEAD 2>/dev/null); b=${b#origin/}; b=${b:-master}; git checkout \"$b\" && git pull --ff-only && git fetch --prune && git switch -c \"$1\"; }; f"
	save = "!f() { git add -A && git commit -m \"$1\"; }; f"
	ship = "!f() { git add -A; git diff --cached --quiet || git commit -m \"${1:-update}\"; git push -u origin HEAD && gh pr create --fill; }; f"
	land = "!f() { gh pr merge --squash --delete-branch; b=$(git symbolic-ref --short refs/remotes/origin/HEAD 2>/dev/null); b=${b#origin/}; b=${b:-master}; git checkout \"$b\" && git pull --ff-only && git fetch --prune; }; f"
	sync = "!f() { b=$(git symbolic-ref --short refs/remotes/origin/HEAD 2>/dev/null); b=${b#origin/}; b=${b:-master}; git checkout \"$b\" && git pull --ff-only && git fetch --prune && git for-each-ref --format='%(refname:short) %(upstream:track)' refs/heads | while read n t; do [ \"$t\" = '[gone]' ] && git branch -D \"$n\"; done; true; }; f"
```

`git ship` и `git land` используют `gh` — без него `start`/`save`/`sync` работают, а PR
создаётся/мержится кнопкой на GitHub. `git land` рассчитан на self-merge (когда ревьюит
сам автор); при ревью напарником он жмёт *Merge* на GitHub, а автор потом делает `git sync`.

## Команды ассистенту (обычными словами)

Не обязательно запускать процесс руками — можно поручить его AI-ассистенту (Claude Code)
простыми словами. Слово **`git`** в фразе — обязательный маркер: значит, речь именно
о GitHub Flow, а не об обычной правке.

| Скажи ассистенту | Что он запускает | Шаг |
|------------------|------------------|-----|
| **git старт** [что делаем] | `git start feature/…` | 1. начать задачу: свежий `master` + новая ветка |
| **git сохрани** | `git save "…"` | 2. закоммитить текущее (можно много раз по ходу) |
| **git отправь** | `git ship` (push + PR) | 3. выложить ветку и открыть Pull Request |
| **git залей** | `git land` | 4. смёржить PR + уборка — **только по этой прямой команде** |
| **git обнови** | `git sync` | 5. подтянуть `master` + снести мёртвые локальные ветки |

Ассистент сам придумывает имя ветки и сообщения коммитов (на английском) — тебе достаточно
описать задачу словами. **Мерж (`git залей`) он делает только по прямой команде** — это
твой «ОК» на попадание кода в `master`; сам, без спроса, не вливает.

Синонимы понимаются по смыслу: «git новая задача» (старт), «git коммит» (сохрани),
«git на ревью» / «git PR» (отправь), «git мержи» (залей), «git синхрон» / «git подтяни» (обнови).

## Часть 2. Одновременная работа и конфликты

Классическая ситуация: двое взяли одинаковый `master`, сделали разные задачи. Первый уже
влил свою ветку. Перезапишет ли второй его изменения при своём PR?

**Нет.** Git защищает код от случайной перезаписи. Возможны два исхода:

| Сценарий | Когда | Что делает GitHub |
|----------|-------|-------------------|
| **Конфликта НЕТ** (авто-слияние) | Правили разные файлы или разные строки | Видит, что изменения не пересекаются, — сливает одной кнопкой |
| **КОНФЛИКТ** | Оба изменили одну и ту же строку одного файла | Блокирует слияние — второму нужно разрешить конфликт локально |

`git merge master` означает: «влей изменения из `master` в **текущую** ветку». Саму ветку
`master` эта команда не трогает.

**Алгоритм разрешения конфликта** (у второго разработчика):

```bash
git checkout master && git pull        # 1. подтянуть чужой код в локальный master
git checkout feature/моя-задача         # 2. вернуться в свою ветку
git merge master                        # 3. вызвать конфликт локально
# 4. Открыть конфликтующий файл. Git разметит проблемное место:
#    <<<<<<< HEAD
#    ваши изменения
#    =======
#    изменения из master
#    >>>>>>> master
#    Убрать маркеры, объединить код правильно, сохранить.
git add .
git commit -m "fix: resolve merge conflict with master"
git push origin feature/моя-задача      # 5. PR снова станет зелёным
```

## Правило синхронизации

Всегда стартуй задачу с самой свежей главной ветки: `git checkout master && git pull`.
Так ты сразу видишь чужие изменения и уменьшаешь шанс конфликта.

## Словарь терминов

- **Репозиторий (repo)** — папка проекта, историю которой отслеживает Git. *(Сейф с документами.)*
- **Ветка (branch)** — изолированная копия проекта для безопасной разработки. *(Параллельный мир.)*
- **Коммит (commit)** — фиксация изменений локально с текстовым описанием. *(Точка сохранения.)*
- **Push (пуш)** — отправка локальных коммитов на GitHub. *(Загрузка в облако.)*
- **Pull (пул)** — скачивание обновлений с GitHub к себе. *(Обновление ленты.)*
- **Pull Request (PR)** — запрос на проверку кода перед слиянием с `master`. *(Сдача работы на проверку.)*
- **Ревью (review, code review)** — просмотр кода из PR вторым человеком перед слиянием: проверить логику, поймать ошибки, при необходимости оставить комментарии и попросить правки. *(Проверка работы перед сдачей.)*
- **Merge (мёрж, слияние)** — объединение изменений из ветки в `master` кнопкой *Merge* на GitHub, после ревью. Именно так код попадает в главную ветку. *(Вклеить готовую задачу в основную версию.)*

## Роль AI-ассистента (Claude Code)

Claude Code следует этому же процессу. Его рабочий регламент (жёсткие правила,
которые он обязан соблюдать) продублирован в разделе **`## Git workflow`** файла
[CLAUDE.md](CLAUDE.md) — потому что автоматически ассистент читает именно `CLAUDE.md`.
Этот файл (`CONTRIBUTING.md`) — **источник истины по процессу** для людей и ассистента;
при расхождении опираемся на него. Ассистент **открывает PR, но никогда не мержит** —
слияние остаётся ручным действием человека после ревью.
